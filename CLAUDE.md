# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Reqcore is an open-source, self-hostable ATS (applicant tracking system) built as a **Nuxt 4** monolith (Vue 3 SSR + Nitro server). Client code lives in `app/`, server code in `server/`, shared code in `shared/`. Runs managed on Railway and self-hosted via Docker Compose.

`ARCHITECTURE.md` is the authoritative deep-dive on architecture, security boundaries, and deployment — read it for anything not covered here. Note it predates several shipped features (AI config, SSO/OIDC, calendar, chatbot, custom properties, comments, activity log), so trust the code over the doc on those.

## Commands

```bash
npm run dev              # Dev server on :3000 (needs Postgres + MinIO running — see below)
npm run build            # Production build → .output/
npm run db:generate      # Generate SQL migration from schema changes in server/database/schema/
npm run db:migrate       # Apply migrations (also auto-runs on server startup)
npm run db:push          # Push schema directly (dev only, skips migration files)
npm run db:studio        # Drizzle Studio DB browser
npm run db:seed          # Seed demo org (demo@reqcore.com / demo1234)
npm run db:reseed        # Delete demo org + reseed

npx vitest               # Run unit tests (watch); `npx vitest run` for one-shot; `npx vitest run tests/unit/rate-limit.test.ts` for one file
npm run test:e2e         # Playwright E2E (boots its own dev server on :3333)
npx vue-tsc --noEmit     # Typecheck (no dedicated lint/typecheck script exists)
```

Local infra (Postgres + MinIO + optional Adminer) comes from Docker Compose: `docker compose up -d`. The dev server needs these running. Env is generated by `./setup.sh` (writes `.env`).

There is **no ESLint config or lint script** — match surrounding style. There is no `test` npm script; unit tests run via `npx vitest` directly.

## Non-obvious conventions (get these wrong and you break things)

**Tenant isolation is the #1 invariant.** Every domain table has `organizationId`. Every query MUST filter `eq(table.organizationId, orgId)` where `orgId` comes from the session — **never** from request body/query/params. This is the core security model.

**Gate every authenticated API route with `requirePermission`, not `requireAuth`.** `requirePermission(event, { candidate: ['read'] })` is the current standard guard — it authenticates, checks an active org is selected, AND enforces RBAC (deny-by-default) against `shared/permissions.ts`. It returns the session; extract `session.session.activeOrganizationId`. `requireAuth` still exists but new routes should use `requirePermission`.

**Server env access goes through `env`, never `process.env`.** All env vars are Zod-validated at startup in `server/utils/env.ts`; the server fails fast on missing/malformed vars.

**`server/utils/` is auto-imported by Nitro** — `db`, `auth`, `env`, `requirePermission`, `uploadToS3`, etc. are globally available with no import statement. Same for `app/composables/` and `app/components/` on the client.

**During SSR, forward cookies manually.** Authenticated `useFetch` calls need `headers: useRequestHeaders(['cookie'])` or the session won't reach internal API calls.

**Document access is always server-proxied** — file bytes stream through authenticated Nitro endpoints (`/api/documents/:id/download`, `/preview`). Never expose presigned S3 URLs or the internal `storageKey` in API responses. Uploads validate MIME via magic bytes (`file-type`), sanitize filenames, and preview is PDF-only.

**Nitro dynamic-segment naming must be consistent.** If `[id].get.ts` and a `[id]/` subdir coexist at one level, both must use the same param name (`[id]`, not `[id]` + `[jobId]`) or the router 404s. See ARCHITECTURE.md §7–8 for the Nuxt page-nesting analog.

**`server/database/schema/auth.ts` is Better Auth's tables — do not modify.** Domain tables live in `app.ts`, SSO in `sso.ts`.

## Code layout

- `server/api/**` — REST routes by filename convention (`index.get.ts`, `[id].patch.ts`). Public/unauthenticated routes live under `server/api/public/`.
- `server/utils/schemas/` — shared Zod schemas; validate input via `readValidatedBody` / `getValidatedQuery`.
- `shared/` — code used by both client and server: `permissions.ts` (RBAC statements), `properties.ts`, `status-transitions.ts`, `feature-flags.ts`, `system-templates.ts`, `chatbot.ts`.
- `app/composables/` — one `useXxx` per domain concern (`useJobs`, `useCandidates`, `useDocuments`, …) wrapping `useFetch`/`$fetch`.
- `tests/unit/*.test.ts` — Vitest. `tests/setup.ts` stubs auto-imported globals (e.g. loggers) since they don't exist outside the Nitro runtime.
- `e2e/critical-flows/*.spec.ts` — Playwright, run **sequentially with shared state** (auth → create job → apply), single worker.

## Data model

`organization` → `job` (draft→open→closed→archived) → `application` (new→screening→interview→offer→hired/rejected) → `candidate` → `document` (in S3). Candidates are deduped per-org by email. Custom `properties` attach to entities (candidates/jobs) and are queried via `server/utils/properties.ts` helpers.

## Deploy

Push to `main` → Railway auto-builds (`npm run build`) and deploys (`node .output/server/index.mjs`). Migrations auto-apply on startup via `server/plugins/migrations.ts` (guarded by a Postgres advisory lock for multi-instance safety). Commits require DCO sign-off (`git commit -s`); PRs fail CI without it.
